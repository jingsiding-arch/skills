# 原文风格继承规则

这个 skill 不是“参考旧文档另写一版”，而是“以旧文档为底稿改出新版”。

## 1. 把原文当成功能描述示例库

补写新能力时，先观察旧文中同类段落怎么写，再照着那个范式写。重点继承：

- 常用句式
- 段落长度
- 术语习惯
- 信息展开顺序
- 表格说明粒度

例如，若原文常用：

- “支持增删改查，可维护……包括……支持批量导入和导出……”
- “可查看……支持……可导出……”
- “系统支持……可实现……便于……”

则新增内容优先沿用这些骨架，而不是突然换成：

- “该模块主要用于……底层通过……实现……”
- “从业务闭环视角看……”
- “平台赋能……提升用户体验……”

## 2. 功能方案的继承重点

功能方案通常更像产品能力说明，重点继承：

- 建设目标写法
- 模块介绍节奏
- “支持 / 可维护 / 可查看 / 可配置 / 可导入导出”等措辞偏好
- 是否偏概述 + 分点，还是偏连贯描述

禁止把原本偏产品说明的方案，改写成研发设计文档、架构说明书或宣传稿。

## 3. 投标参数的继承重点

投标参数通常更像应答文档，重点继承：

- 参数项应答口径
- “支持 / 支持配置 / 支持对接 / 支持管理 / 需确认”等风险表达
- 限制条件表达方式
- 原表头和列顺序

禁止出现以下问题：

- 把原本谨慎的参数应答，改成夸张宣传
- 把“支持配置后实现”写成“默认支持”
- 把“可对接”写成“已深度集成”

## 4. 新增段落的最低标准

新增段落至少满足：

1. 单独读起来像原文作者写的
2. 插回原章节后不突兀
3. 与相邻两段的粒度相近
4. 不引入原文没有的陌生表达体系

## 4.5 新增内容的插入位置规则

新增内容不仅要“像原文”，还要“放在原文最合理的位置”。

优先级如下：

1. **原文已有对应父章节**：补到该父章节下面最近的层级
2. **原文已有近似章节但层级不足**：在该章节下新增同级小节
3. **原文完全没有对应章节**：再新增新的一级/二级章节

禁止把本应属于不同原章节的新增页面，统一塞到文末补充章。

还要注意：

- 原文已有章节的先后顺序不能被打乱
- 如果原文已有 `A -> B -> C` 的顺序，新增内容不能把 `C` 挤到 `B` 前面
- 对带编号文档，新增编号必须保持顺序正确
- 非业务性的说明项不要抬成与业务章节同级的目录标题，避免把“说明”“遍历说明”“生成说明”这类内容混入主目录

例如：

- “集中住宿床位管理 / 集中住宿变动记录 / 集中住宿房间统计”应放在“集中住宿管理”下
- “宿舍公告管理”应优先放在“宿管大屏”或最贴近的信息发布章节下
- “排宿操作记录”应放在“学生排宿管理”下，而不是独立飘在文末

## 5. 建议写法

优先做这三件事：

- 复制原文最近似段落的表达骨架
- 只替换能力点、字段项、流程环节、统计维度
- 保持原文相同的信息展开顺序

## 6. 最终自检

交付前问自己：

- 这一段像不像原文？
- 如果删掉高亮，用户能否一眼看出这是后补的“另一种文风”？
- 方案和参数是否都延续了各自原有的语气和口径？
