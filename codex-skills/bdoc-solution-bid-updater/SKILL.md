# B-side Solution + Bid Parameter Updater

把“旧版功能方案 + 旧版功能投标参数 + 最新变更信息”整理成两份互相对齐、且**直接继承原文风格**的新版完整文档。

## Route boundary

优先用于以下场景：

- 用户明确要**同时**更新《功能方案》和《功能投标参数》
- 用户要求“方案和投标参数口径保持一致”
- 用户已有旧文档，希望在**不推翻原模板**的前提下升级到最新版
- 用户要求保留原编号、术语、表格结构、正式程度和投标口径

不用于以下场景：

- 从零设计新模块 PRD，而不是更新旧文档
- 只做操作手册、培训手册或纯页面文案润色
- 只要技术架构设计、接口设计、数据库设计
- 需要法律、合规、资质承诺类正式审定的内容替代人工审核

## Default assumptions

- 默认输出中文
- 默认以“旧文档为底稿”直接补写，最终输出**完整新版文档**而不是只输出增量片段
- 默认保留原版已有正文、表格和图片，不删除、不替换、不改写原文既有内容；若需调整，以“原文保留 + 补充说明”优先
- 默认优先做“局部增补”，交付时要把补写结果合并回原结构，形成可直接替换旧版的整篇新文档
- 默认功能方案与投标参数需要**双向校对**，避免一边更新、一边遗漏
- 默认把旧文档当作该产品的“功能描述示例库”，新增内容必须尽量复用其叙述句式、术语偏好、段落组织和说明粒度
- 默认新增/修改内容需要显式标记；非 docx 场景优先使用 `<mark>...</mark>`
- 默认投标参数采用**保守应答口径**：禁止把“可配置 / 可定制 / 规划中”写成“已原生支持”

## Minimum inputs

至少需要三类输入：

1. 旧版《功能方案》
2. 旧版《功能投标参数》
3. 最新变更依据
   - 可为 PRD、需求说明、版本说明、页面更新说明、会议纪要、补充备注等

可选补充：

- 目标版本号 / 日期
- 客户行业或项目背景
- 是否严格按投标口径输出
- 是否允许 AI 做低风险补全
- 高亮方式要求（底色、修订、红字、括号标识）

如果只提供了其中一份旧文档，允许继续，但必须显式说明：**另一份文档只能做推测性草拟，跨文档一致性会下降。**

## Compact workflow

### 1. 先建“变更映射矩阵”

开始写正文前，先把最新变化拆成结构化条目。至少回答：

- 变更点是什么
- 依据来自哪里
- 影响功能方案的哪些章节
- 影响投标参数的哪些参数项/表格行
- 当前能力状态是“已支持 / 支持配置 / 需二开 / 待确认”中的哪一种

先读取 `references/update-matrix-template.md`。

### 1.5 最新变更依据若来自网页后台，必须做“菜单 + 按钮”双层遍历

如果最新变更依据来自已登录的 B 端后台，不要只停留在左侧菜单或列表页标题。

至少要补做以下动作：

- 进入对应菜单页
- 读取列表表头、筛选项、操作列
- 点击**非破坏性按钮**进一步确认能力边界，例如：
  - `新增`
  - `编辑`（仅打开，不提交）
  - `详情`
  - `查看`
  - `复制`
  - `统计`
  - `导出配置`
  - `公共设置`
  - `设置`
  - `点击管理`
- 打开弹窗、抽屉或表单后，记录：
  - 字段项
  - 分组标题
  - 可配置规则
  - 可选项
  - 按钮文案
  - 流程环节

禁止点击以下高风险按钮：

- `删除`
- `批量删除`
- `确认`
- `提交`
- `保存`
- `启用/停用`
- 任何会真正写入、发送、清空、下线、归档的数据动作

如果按钮打开的是配置弹窗或详情弹窗，优先在关闭前提下读取结构信息，把它当作功能边界证据。

### 1.6 网页后台核查时的经验性回退策略

做已登录后台调研时，补充遵循以下经验规则：

- **直达 URL 失败不等于功能不存在**：如果你手工猜的路由打开后出现 `404`、`OOPS`、无权限提示，不能直接判断该功能不存在。优先回到**已登录页面左侧菜单点击进入**，以菜单真实跳转结果为准。很多后台存在：
  - `kebab-case / camelCase` 混用
  - 有的页面带 `/list`，有的页面不带
  - 同类模块命名不完全对称
- **菜单命中后，再记录真实 URL**：技能交付时，若页面 URL 与猜测不一致，应以菜单点击后实际进入的 URL 作为依据，并在变更摘要或备注中保留“经菜单核实”的说明。
- **页内二级视图也算能力边界**：如果进入页面后存在页内切换、标签页、统计卡片、视图切换（例如“个人违纪记录 / 宿舍违纪记录”），要把它们视为同一父模块下的二级能力，而不是漏掉。
- **统计页没有新增按钮也要完整取证**：对于统计类页面，除了常规按钮和表头，还要记录：
  - 顶部 KPI/统计卡片名称
  - 统计维度名称
  - 列表中的“查看明细 / 查看统计 / 查看记录”等下钻入口
- **空数据页也要留证**：即使当前页面“暂无数据”，仍要记录：
  - 筛选项
  - 表头
  - 顶部操作按钮
  - 页内统计卡片/分布项
  这些信息足以支撑方案和投标参数中的能力描述。

### 1.7 页面证据提炼为文档内容时的写法规则

从后台页面证据落到《功能方案》与《功能投标参数》时，优先按以下顺序提炼：

1. **页面定位**：该页属于哪个父模块，是否应写成新增小节
2. **管理对象**：该页管理的核心对象是什么（如批次、记录、公告、设备、整改项、评比结果）
3. **查询维度**：筛选项反映了哪些业务维度（如校区、楼栋、院系、时间段、状态）
4. **结果展示**：表头、统计卡片、明细项反映了哪些可查看信息
5. **操作能力**：按钮和操作列反映了哪些真实能力边界（如新增、导出、复制、继承、批量复核、查看明细）
6. **流程/规则**：新增弹窗、编辑抽屉、配置表单中有哪些字段、开关、规则项

写入文档时注意：

- 《功能方案》优先写成“支持……可维护……可查看……支持导入导出……”的产品能力说明
- 《功能投标参数》优先写成“需支持……”的应答句式，并明确前置条件和边界
- 不要把页面上出现的每一个控件都机械罗列进正文；应抽象为**稳定的业务能力描述**
- 若只核到了列表、筛选、表头，但没成功打开弹窗，则只能写到列表能力和统计能力，不要臆造表单字段

做网页后台核查类任务时，建议同时读取 `references/backend-research-playbook.md`，尤其适用于：

- 需要从真实后台页面回填方案和投标参数
- 直达 URL 失败，需要改走菜单核查
- 页面暂无数据，但仍需提炼可写入文档的能力
- 统计页、页内双视图、查看明细类页面较多的场景

如果任务和“宿舍检查 / 检查批次 / 检查记录 / 整改记录 / 评比记录 / 统计页”这类后台页面相似，可进一步读取 `references/case-dorm-inspection-example.md`，参考：

- 如何从页面证据抽象出稳定能力
- 如何分别写成“方案口吻”和“参数口吻”
- 哪些内容可以安全写入，哪些只能保留为待确认项

### 2. 更新功能方案

先把旧版方案当作底稿，按原结构逐段判断“保留 / 增补 / 新增章节”，不要另起一份全新写法。优先更新这些位置：

- 背景 / 目标 / 范围
- 业务场景
- 页面或流程说明
- 业务规则
- 字段定义
- 权限边界
- 异常与限制
- 验收要点

能补入原章节就补入原章节；原文没有时，再按原文风格新增同级章节。新增段落时，优先复用原文里已经出现过的表述骨架，例如“支持……可维护……包括……支持导入导出……”这类句型。

如果原文已经存在对应模块，则**必须优先补到该模块下面最近的层级**，不要把它们统一塞到文末“补充说明”或“现网能力补充”中。  
例如：

- 原文已有“集中住宿管理”，则“集中住宿床位管理 / 集中住宿变动记录 / 集中住宿房间统计”应补到该章节下，作为新增小节
- 原文已有“查寝”，则“查寝结果统计 / 特殊打卡审核 / 蓝牙信标管理”应优先补到“查寝”章节下
- 只有当原文完全没有对应父章节时，才允许新增新的一级或二级章节

同时，**不得打乱原有章节顺序**：

- 原版已有的章节顺序必须保持不变
- 新增同级章节若无对应父章节，应插入在合理的新位置，但不能把原有章节挤到错误顺序
- 对带编号的文档，新增编号必须保证顺序递增，禁止出现 `2.13` 在 `2.12` 前面

### 3. 更新投标参数

先把旧版投标参数当作底稿，逐项检查受影响参数，保持原表头、原列顺序、原口径。默认优先“保留旧参数原文 + 在对应位置增补新参数项或补充说明”。重点写清：

- 满足情况
- 响应说明
- 前置条件
- 限制条件
- 是否需要配置 / 定制 / 二开

如果最新需求没有足够证据支撑“完全满足”，不要强写“满足”。若旧版参数已有成熟表达，优先沿用其应答句式，只替换受影响的能力描述和限制条件。

### 4. 做跨文档一致性校验

功能方案和投标参数必须互相对齐：

- 方案里新增的能力，投标参数不能漏掉
- 投标参数里承诺的能力，方案里要能找到支撑描述
- 名称、角色、前置条件、限制说明、边界口径要一致

写作前或交付前，读取 `references/consistency-checklist.md` 完整核一遍。

### 5. 做原文风格继承校验

正式交付前，读取 `references/style-inheritance-rules.md`，至少核对：

- 新增段落是否像原文，不像 AI 另写的一套
- 同一章节内句式长短、术语偏好、功能说明粒度是否与原文接近
- 是否把原文的“产品方案口吻”误写成“研发设计说明”
- 是否把原文的“投标响应口吻”误写成“宣传文案”

### 6. 输出交付物

默认输出 5 部分：

1. 新版《功能方案》完整文档
2. 新版《功能投标参数》完整文档
3. 变更映射表
4. 变更摘要
5. 待确认项 / AI 补全假设

如果用户只要正文，可省略 3-5，但高风险缺口仍必须提醒。

## Hard rules

- 不得虚构已上线、已交付、已有案例、已过认证等事实
- 不得把“可支持”偷换成“已支持”
- 不得把投标响应写成营销文案
- 不得因为赶时间而跳过双文档一致性检查
- 不得 silently 改写原文关键术语、编号体系、表格口径
- 不得脱离原文范式，写出与旧文风格明显不一致的新段落
- 不得删除原版已有正文内容
- 不得删除、替换原版已有图片
- 若最新依据来自网页后台，不得只看菜单名就下结论；必须至少抽样点击关键非破坏性按钮确认字段和能力边界
- 不得为了省事把多个已有模块的新增页面统一塞到文末补充章；应按最近原章节回填
- 不得因为插入新增章节而打乱原有章节顺序或造成编号倒挂

## High-risk items that must stay explicit

以下信息若没有明确依据，必须进入“待确认项”，不能直接写成既成事实：

- 性能指标、并发量、容量、时效 SLA
- 外部系统接口与对接深度
- 权限边界与数据隔离级别
- 信创、等保、合规、行业资质
- 是否已有正式上线客户案例
- 是否完全满足投标硬性条款

## Recommended output style

- 非 docx 交付默认用 `<mark>新增或修改内容</mark>`
- 对新增章节、改写句、增补表格行分别做局部标记，不整篇泛黄
- 每个“待确认项”都尽量写清：缺什么、影响什么、建议谁确认
- 当用户要求“生成两个新的文档”时，默认输出**完整合并版**，而不是只给增量 patch
- 若原始输入为 docx，优先生成“保留原文与原图、只插入新增内容”的新版 docx
- 非业务性的说明项、遍历说明、生成说明、AI 提示等默认不要作为正文目录层级标题；优先降级为普通段落、文末说明或附录说明，避免污染主目录

## Example requests

- “基于旧版功能方案和旧版投标参数，把这个模块的新需求同步更新成最新版。”
- “我有一份老方案和老投标参数，再加这次迭代说明，帮我联动生成新的两份文档。”
- “不要重写模板，沿用原结构，把新增功能补到方案和投标参数里，并列出待确认项。”
- “在原方案和原参数文档的基础上直接改，保持原文写法，最后生成两个新的完整文档。”
