# 路由决策表

用于快速判断当前需求该走哪个 skill，以及本 skill 内该选哪套模板。

## 1. 先判 skill

| 场景特征 | 建议路由 | 原因 |
| --- | --- | --- |
| 单页面、小优化、局部规则、单个配置项、单张列表、单个流程断点 | `bdoc-optimization-testcase-writer` | 目标是快速收敛方案、评审点和测试用例 |
| 从 0 到 1 新增完整模块 | `module-prd-writer` | 需要完整字段、状态、权限、交互、验收设计 |
| 多角色、多系统、多阶段复杂闭环 | `module-prd-writer` | 已超出“优化需求”粒度 |
| 明显需要完整字段表、权限矩阵、状态机、接口协同 | `module-prd-writer` | 研发交付粒度更重 |
| 用户只想挑战现有思路、判断值不值得做 | `bdoc-optimization-testcase-writer` | 先做轻量诊断和替代方案更合适 |

## 2. 再判模板

| 场景特征 | 建议模板 | 说明 |
| --- | --- | --- |
| 字段、按钮、开关、提示文案、列表展示、小规则 | `assets/optimization-doc-template-light.md` | 默认不保留流程和权限小节 |
| 跨页面、跨角色、跨状态、审批链路、批量处理、异常分支、回退路径 | `assets/optimization-doc-template.md` | 支持按需保留流程和权限小节 |

## 3. 歧义时的保守决策

1. 如果删掉 `操作流程` 后，研发、测试、评审理解不受影响，就按轻需求处理。
2. 如果用户真正担心的是目标跑偏、方案过重、责任边界，而不是字段细节，优先留在本 skill 做挑战。
3. 如果开始出现完整模块边界、权限矩阵、状态机、跨系统协同，尽快切到 `module-prd-writer`，不要硬撑。
