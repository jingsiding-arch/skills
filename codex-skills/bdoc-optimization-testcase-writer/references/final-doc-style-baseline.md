# 飞书定稿样式基线

基于用户最终确认并手工调整后的飞书文档，后续默认按这套风格生成优化需求文档。

## 一、整体原则

- 默认生成的是**轻量优化方案**，不是大而全 PRD。
- 目标是：一眼能看懂、能评审、能落地、能测试。
- 保持结构稳定、标题固定、段落简短。
- 默认不写多余的分析过程、协作痕迹、问题定义、方案对比、影响范围、验收口径。
- 默认不写“原来的 / 优化后”对比，除非用户明确要求保留对照。

## 二、固定章节

最终飞书文档默认只保留以下顶层结构：

1. `需求背景`
2. `实现方案概览`
3. `需求方案描述`
4. `测试用例`
5. `当前文档下的待确认项`

其中：

- `测试用例` 下固定拆成：
  - `4.1 建议优先回归场景`
  - `4.2 具体测试用例`

## 三、各章节写法

### 1. 需求背景

- 使用一个**克制的蓝色高亮块**承载。
- 内容不要压成一句话，也不要写成长篇背景介绍。
- 默认至少包含三层信息：
  - 当前业务场景
  - 当前问题 / 影响
  - 为什么现在需要做

推荐形态：
- 先用 1 句交代业务对象
- 再用 1 段交代问题和影响
- 最后用 1 句收束到本次优化目标

### 2. 实现方案概览

- 使用一个**克制的蓝色高亮块**承载。
- 只写方案核心结论，不展开冗长论证。
- 适合用短 bullet 列出：
  - 入口
  - 开关/规则
  - 执行口径
  - 历史数据处理
  - 关键结果

### 3. 需求方案描述

默认必须按**页面维度**展开。

页面级标题结构严格参考用户确认的飞书文档：

```md
## 需求方案描述
### 2.1 页面名称 / 页面路径
#### 2.1.1 页面 / 交互
#### 2.1.2 逻辑 / 规则
#### 2.1.3 操作流程
#### 2.1.4 权限控制
```

如果有多个页面，则继续扩展为：

```md
### 2.2 页面名称 / 页面路径
#### 2.2.1 页面 / 交互
#### 2.2.2 逻辑 / 规则
#### 2.2.3 操作流程
#### 2.2.4 权限控制
```

每个页面至少保留 `页面 / 交互`、`逻辑 / 规则`。`操作流程` 和 `权限控制` 不为凑结构机械保留，只在确实影响理解、实现或测试时输出。

如果页面属于审核、审批、待办、确认等通常带移动端的功能，页面级标题默认同时标记端别：

```md
### 2.2 学籍异动审核（PC） / 学籍异动审批（移动端）
```

若移动端页面名未确认，也要写出端别占位：

```md
### 2.2 学籍异动审核（PC） / 移动端待确认（移动端待确认）
```

各子标题职责：

#### 页面 / 交互

- 写页面展示、入口位置、按钮状态、弹窗内容、提示语、交互动作。
- 优先描述用户最终会看到和操作到的内容。
- 凡涉及字段填写、表单录入、配置维护，必须把字段口径写清：
  - 至少标记 `是否必填`
  - 已知时标记 `默认值`
  - 推荐写法：`字段名（必填，默认值：当天）`
  - 默认值未知但很关键时，写成 `默认值：待确认`
- 字段较多时，可直接改成简表：
  - `字段 | 是否必填 | 默认值 | 说明`

#### 逻辑 / 规则

- 写业务规则、执行口径、判定条件、批量处理逻辑、异常跳过逻辑。
- 条理上优先放“开关开启后 / 开关关闭后 / 特殊情况”。

#### 操作流程

- 仅当需求涉及跨页面跳转、跨角色协同、跨状态流转、审批链路变化、批量处理链路、异常分支或回退路径时输出。
- 先用 1 句简短说明流程。
- 需要输出时默认使用**飞书白板流程图**或流程图组件。
- 正确完成标准是：
  - 文档中实际插入了 whiteboard / 流程图组件
  - 组件内已写入流程图内容
  - 不是只在正文里保留一句流程说明
- 不再用长编号列表代替最终流程展示，除非用户明确说不要流程图，或当前环境无法创建白板并已说明原因。无法落图时保留 Mermaid 草稿或 `流程图待落白板` 标记，不能声称已经插入白板。

#### 权限控制

- 仅当本次新增按钮/入口/操作能力、改变可见范围/可操作范围，或角色边界影响实现与测试时输出。
- 只写该页面按钮或入口相关权限。
- 不写全模块权限矩阵。

## 四、测试用例写法

### 4.1 建议优先回归场景

- 固定放在 `测试用例` 下的第一个小节。
- 用 2-4 条编号列表写最优先回归范围。

### 4.2 具体测试用例

- 默认使用表格。
- 列固定为：
  - `用例类型`
  - `操作步骤`
  - `预期结果`

- `前置条件` 默认并入 `操作步骤`；除非用户明确要求，不单独保留 `用例标题`、`前置条件` 两列。
- 默认按 `主链路 / 关键分支 / 异常场景 / 边界操作` 拆成多张小表，每张表 1-3 条。
- 用例类型默认覆盖：
  - 主链路
  - 关键分支
  - 异常场景
  - 边界操作

## 五、待确认项写法

- 顶层标题固定为：`当前文档下的待确认项`
- 只保留真正还没定的 1-3 条问题。
- 不要把已经确认的内容继续留在待确认项里。

## 六、默认排版要求

- 允许轻美化，但必须克制。
- 推荐：
  - `需求背景` 用蓝色高亮块
  - `实现方案概览` 用蓝色高亮块
  - 必要时用分割线分隔大章节
  - `测试用例` 用飞书表格
  - 需要展开 `操作流程` 时用白板流程图
  - 重点关键词允许少量加粗
  - 如需颜色强化，仅使用蓝色、紫色两色
- 飞书兼容性安全规则：
  - 不使用 `> - xxx`、`> 1. xxx` 这类 `blockquote + list` 组合
  - 不嵌套 `callout`
  - 如果正文已手写 `callout`，创建文档时优先用更保守的参数，避免自动美化再套一层高亮块

- 重点样式建议：
  - 开关名、模块名、字段标签、端别标记可优先 `**加粗**`
  - 若当前写入路径颜色稳定，可对少量标签使用蓝色或紫色
  - 不要整段大面积着色，不要把整个列表染色

- 默认不要：
  - 顶部导读
  - 阅读导航
  - 结构导图
  - 来源代码块
  - 截图占位
  - 大段 AI 解释文字
