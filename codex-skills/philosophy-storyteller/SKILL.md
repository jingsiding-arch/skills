# Philosophy Storyteller

把一句抽象的小道理，转成一个像朋友夜里聊天一样展开的中文小故事：对话推进、情绪先行、隐喻递进，最后留下一句安静但有穿透力的落点。

## Route Boundary

优先用于以下场景：

- 用户给出一个小道理、人生哲理、情绪洞察、关系观察、心理概念或价值判断
- 用户希望“讲成故事”“写成小故事”“写成对话”“不要说教”
- 用户提到“夏令营效应”“烟花和灯”“火柴和太阳”这类隐喻式故事感
- 用户希望把抽象观点变成适合阅读、传播、共鸣的中文短篇

不用于以下场景：

- 只要概念定义、论文解释、知识科普或严格事实考据
- 要求完整小说、世界观、多人剧情、强戏剧冲突或爽文反转
- 要求儿童寓言、品牌营销故事、广告脚本或短视频分镜
- 要求模仿特定在世作者的风格

## Minimum Inputs

至少需要一个输入：

- `core_truth`: 一句小道理、哲理、感悟或抽象观点

可选输入：

- `scene_hint`: 希望发生在哪类关系或场景中，例如朋友、恋人、职场、亲子、独处、旅途
- `tone`: 更温柔、更冷静、更锋利、更治愈、更克制
- `length`: 短篇、标准、长篇
- `metaphor_seed`: 用户指定的核心隐喻，例如雨伞、候车厅、旧钥匙、潮汐、借来的房间

如果用户只给一句道理，直接开始创作，不要先追问。

## Compact Workflow

1. 提炼道理的核心矛盾
   - 把道理拆成“人以为 A，但现实常常是 B”。
   - 找到里面的情绪：困惑、遗憾、羞耻、执念、释然、松手、重新理解。

2. 选择一个普通但有刺的生活场景
   - 场景要低戏剧、强共鸣，例如聊天冷掉、离职、搬家、成年后和父母相处、朋友渐远、努力没有回声。
   - 不要用宏大事件开场。

3. 让两个人用问答把故事问出来
   - 默认采用双人对话，不写旁白或少写旁白。
   - 一个角色负责困惑和抵抗，另一个角色负责慢慢命名、拆解、安放。
   - 问句要短，回答要像真实聊天，不要像演讲。

4. 建立核心隐喻
   - 隐喻必须来自场景或情绪，不要硬套漂亮词。
   - 可以给它一个温和的名字，但如果是临时创造的概念，要说“我把它叫作……”而不是伪装成真实学术概念。
   - 让隐喻至少递进两次：先解释，再被质疑，再换一个更准确的比喻落地。

5. 保留情绪抵抗
   - 不要太快治愈主角。
   - 让主角说出“不甘心”“那算什么”“是不是我错了”“可我还是想要”等真实反应。
   - 回答不能否定痛苦，只能重新放置痛苦。

6. 以安静的句子收束
   - 结尾不要喊口号，不要总结全文。
   - 最后一两句应该像轻轻放下，而不是宣布胜利。

## Style Rules

创作前先读取 `references/story-pattern.md`，交付前用 `references/quality-checklist.md` 自查。

默认输出规则：

- 使用中文。
- 标准篇幅约 1200 到 2200 个汉字；用户要求短篇时控制在 500 到 900 字。
- 默认使用引号包裹对话，不加角色名。
- 允许少量省略号，但不要滥用。
- 每轮对话尽量短，保持一问一答的节奏。
- 不要把道理直接写成文章段落。
- 不要写“这个故事告诉我们”。
- 不要在结尾解释隐喻。

## Output Format

默认只输出故事正文。

如果用户要求“顺便说明怎么改的”，可以在故事后追加一个极短说明：

- `核心矛盾`
- `使用的隐喻`
- `情绪落点`

否则不要附加拆解说明，保持故事的余味。
