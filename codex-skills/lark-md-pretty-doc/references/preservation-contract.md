# Preservation Contract

这个 skill 的核心不是“美化 Markdown 内容”，而是“**在不改正文的前提下，把 Markdown 稳定搬运到飞书文档**”。

## 允许

- 新建飞书文档标题
- 在不改写正文的前提下新增克制的主题化包装
- 在正文前新增一条分割线
- 把 Markdown 正文按原顺序切片后分次写入
- 当且仅当用户明确点名某个章节时，按标题整段排除该章节
- 当且仅当用户明确要求时，按统一语义规则给部分关键词添加内联颜色

## 不允许

- 改写任意正文字符
- 合并或拆分正文段落内容
- 重命名原始章节
- 调整原始段落顺序
- 替换原始列表符号、编号、表格文本、代码内容
- 删除空行来“优化排版”
- 为了适配 callout 把正文内容搬进去
- 生成顶部导读块或阅读导航
- 在用户未明确要求时擅自删除任意章节
- 在用户未明确要求时擅自给正文做颜色改写

## 关于“显式删章”

这是一条**用户驱动的例外规则**。

正确做法：

- 用户明确说“删除工程附表”
- 通过标题匹配定位 `# 工程附表`
- 删除从该标题开始，到下一个同级或更高层级标题前的整段内容

错误做法：

- 因为觉得篇幅太长，擅自删附录
- 只删除章节中的一部分行
- 把章节内容挪到别处
- 删除章节后顺手改写其他正文

## 关于“显式语义着色”

这也是一条**用户驱动的例外规则**。

正确做法：

- 用户明确说“优化一下颜色标记”
- 或用户明确说“你自动推荐颜色策略”
- 只按统一语义色系统一处理关键词
- PRD 默认优先选择更克制的 `semantic-conservative`
- 默认跳过代码块、表格和已有富文本标签

错误做法：

- 看着哪里重要就随手上色
- 把整段正文改成彩色
- 同一类语义混用多种颜色
- 在未征得用户同意时改正文颜色

## 关于“分块”

分块只是传输策略，不是内容改写。

正确做法：

- 从原文中按边界切出连续片段
- 片段拼接起来后，必须与原文正文完全一致

错误做法：

- 切块时顺便整理标题
- 切块时补充摘要
- 切块时去掉空行
- 切块时把列表改成表格

## 关于主题包装

默认主题包装只承担三个职责：

- 增强标题区和章节节奏
- 用克制的高亮块承接摘要、版本信息、实现方案、待确认项
- 保持正文仍然是文档主角

它不能替代正文摘要，也不应承载业务结论，更不能把正文重写成另一份内容。
