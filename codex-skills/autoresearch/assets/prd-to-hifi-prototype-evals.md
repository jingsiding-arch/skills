EVAL 1: 直接开做不阻塞
Question: 对于明确要求“直接做/直接开始/不用等确认”的输入，输出是否不把确认动作变成阻塞，并清楚区分已确认信息与假设？
Pass: 在这类 prompt 下，输出包含 Confirmed/Assumptions/TBD 或中文等价标记，并继续给出 manifest 摘要、页面方案或实现步骤，且没有“等确认后再做”的阻塞措辞。
Fail: 仍要求用户先确认 manifest 或先补齐全部材料后再开始，或完全不区分已确认信息与假设。

EVAL 2: 模糊输入先收口
Question: 对于只有模块名的模糊输入，输出是否先给澄清问题、素材清单和页面/流程骨架，而不是伪造完整实施方案？
Pass: 在模糊 prompt 下，输出出现澄清问题或素材清单，并给出页面骨架或流程骨架，同时没有展开成完整的代码接入实施清单。
Fail: 在信息极少时直接写成完整原型实施方案，或没有给出任何澄清与骨架。

EVAL 3: 半成品先出 v0.1
Question: 对于半成品材料，输出是否先推进 v0.1 原型方案，并把追问控制在关键问题内？
Pass: 在半成品 prompt 下，输出明确给出 v0.1、P0 或“先出一版”的推进方案，且追问不超过 3 个或明确声明只问关键问题。
Fail: 只罗列缺失信息不推进方案，或抛出大量追问把原型工作卡住。

EVAL 4: 原型闭环意识
Question: 输出是否覆盖 P0 页面、主流程、异常/边界、导航入口、Mock/状态流转中的至少三项？
Pass: 输出同时体现至少三类与“可演示闭环”直接相关的内容，而不是只停在抽象描述。
Fail: 缺少闭环视角，没有把页面、流程、异常、导航、Mock 等演示要素串起来。

EVAL 5: 视觉与仓库策略贴合上下文
Question: 当 prompt 提到现有 React+Vite/router 或 Figma 高保真时，输出是否分别强调复用现有仓库接入与以 Figma 为视觉真源？
Pass: 在相关 prompt 下，输出明确提到接入现有 React+Vite/router、避免另起项目，以及 Figma、Token、1:1 还原或样式对齐等上下文匹配策略。
Fail: 忽略现有仓库/路由信息，默认重搭项目，或对 Figma 高保真要求没有给出对应视觉还原策略。
