EVAL 1: 页面元素定义是否清晰
Question: PRD 是否把页面元素写清了？
Pass: 筛选项、列表字段、表单项至少写清名称、类型、数据来源、展示或输入方式，研发无需再追问“是数据字典还是写死”“字段怎么显示”。
Fail: 只写“有筛选/有字段/有表单”，但没有写清来源、类型、展示规则或输入规则。

EVAL 2: 交互与触发逻辑是否清晰
Question: 这份 PRD 是否把交互和触发逻辑写清了？
Pass: 关键操作至少写清用户动作、前端反馈、后端触发逻辑、成功结果和失败处理。
Fail: 只有“支持新增/编辑/保存/启用”之类功能描述，没有写清操作后的页面变化和后端行为。

EVAL 3: 生效规则是否清晰
Question: 这份 PRD 是否把全局生效或者局部生效规则写清了？
Pass: 已明确哪些配置全局或局部生效、生效范围、生效时点、影响对象，以及必要时的优先级或冲突处理。
Fail: 配置只在局部功能里提到，没有说明它会影响哪些页面、流程或数据。

EVAL 4: 配置可改边界是否清晰
Question: 这份 PRD 是否把“哪些配置可改、哪些不可改”写清了？
Pass: 对关键配置项明确说明在不同状态下是否可改、为什么限制、改后影响什么。
Fail: 只写“不可编辑”或“谨慎修改”，没有说明限制条件、原因和影响范围。

EVAL 5: 异常与边界处理是否清晰
Question: 这份 PRD 是否把异常和边界情况写清了？
Pass: 至少覆盖关键异常、边界条件、失败处理、兜底方式或回退方式，避免只写主流程。
Fail: 只有主流程，没有配置变更、流程中断、依赖缺失、状态冲突等异常场景的处理规则。

EVAL 6: 是否可直接支持研发与测试验收
Question: 这份 PRD 是否能直接支持研发与测试验收？
Pass: 研发和测试可以仅依据 PRD 理解页面逻辑、规则、生效机制和验收标准，并据此拆任务或写测试点。
Fail: PRD 看起来完整，但研发和测试仍需要大量追问才能确认开发口径和验收条件。
