# 模块需求梳理清单

## 1. 业务背景
- 这个模块为什么现在要做，是政策驱动、业务增长、风险治理还是效率提升？
- 当前流程如何运行，痛点发生在谁身上，损失体现在哪里？
- 有没有明确时间节点、上线窗口、合规要求？

## 2. 目标与边界
- 模块本次必须交付的结果是什么，什么算成功？
- 哪些内容是同一系统里的其他模块负责，不应混进当前 PRD？
- 是否存在一期/二期拆分，哪些需求先做，哪些延后？

## 3. 角色与组织
- 谁发起、谁执行、谁审核、谁查看结果、谁兜底处理异常？
- 是否存在多组织、多校区、多部门、多班级等数据隔离规则？
- 不同角色看到的入口、字段、操作是否不同？

## 4. 流程与规则
- 主流程的起点、终点、关键节点、状态变化是什么？
- 哪些规则影响结果：时间、次数、阈值、频率、冲突校验、自动处理？
- 哪些情况需要撤回、补录、重试、覆盖、人工介入？

## 5. 数据与口径
- 核心对象有哪些，例如任务、记录、审批单、统计报表？
- 字段从哪里来，哪些可编辑，哪些由系统计算，哪些需要留痕？
- 统计口径是否统一，分母分子、时间口径、去重逻辑是否明确？

## 6. 交互与交付
- 模块需要哪些页面、入口、列表、详情、弹窗、导出、消息提醒？
- 有没有移动端、PC 端、大屏、投屏、第三方入口等多端差异？
- 上线后如何验收，哪些场景必须在 PRD 中给出明确标准？
